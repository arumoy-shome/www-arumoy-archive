Notes on privacy

There are a lot of independent, privacy focused companies poping up who are pushing their privacy preserving products. However, there is no guarantee that they will remain independent in the future. There is also the possibility of them getting bought by larger corporations or simply going bankrupt.

There is a trade-off between privacy and utility. One suffers while emphasis is put on the other. I am fully aware that I am paying for the functionality provided by products such as Google with my data. I am okay with this because:

  1. Large companies such as Google/Apple have a lot to lose if they have security breaches
  2. They do not have a history of catastrophic security breaches
  3. They are transparent about how my data is being used and so-far they have not abused my data

Knowing the above, I opt to keep my data centralised within select companies whome I trust, in return for superior functionality.

Research workflow

Here I outline a workflow which I have developed to handle the non-linear nature of scientific research. There are days when one reads several papers without making much sense of them. However, a single paper the next day can result in a clear picture of the domain. The workflow acknowledges the fact that there are many more failures rather than successes in research. However, a failure is still an outcome - and this is what helps us decide what to do next - so it should still be recorded.

There are 3 interdependent phases to this process described below.

Finding papers

I save the bibtex entry in a bib file (generally in the project root along with the other latex files). This is because the bibtex-mode in emacs provides powerful search options, templates for adding entries, sorting entries, cleaning up, and much more. Besides, the bib files ultimately gets used in the report so it makes sense to group the two together and keep them in the same location.

First, I use good ol' Google Scholar (along with a plugin to enable my institution's proxy) to find papers. Once, I find a paper I open it and read the abstract. If the abstract appeals to my current research focus or is interesting to me in general, I download it and save it in a folder which syncs across my devices. Next, I grab the bibtex entry and create a task using org-capture where I save the bibtex entry, one liner on why I found the paper interesting. I add the location to the pdf on disk, I set the title of the task as the bibtex entry and I save it to my inbox (or relevant project org file). I use org-capture along with this template to automate this process.

Reading papers

I read the papers on my laptop. My screen is usually split 50-50 between Emacs (where I take notes) and a pdf viewer. I do not make any highlights because its easy to abuse and it ties the information to the pdf viewer application that I am using.

Instead, I take notes in plaintext (as of 2020, I have been using org-mode, an Emacs major mode to do so). There are several, rationales for doing this digitally. First, I require these notes to be searchable and linkable to other notes. Second, research projects often span for long durations. Having digital notes ensures that I do not have notes distributed amongst several analog papers/journals. Lastly, I take notes on papers in a specific manner. These notes have a specific purpose - and thus a carefully constructed structure as described in the next section - and thus come under the category of hard information (see Categories of Information).

Writing notes

Notes on paper being read will always be in relation to the current research focus/goal. A paper may be read multiple times but will have different notes if they were made with reference to different research questions.

Following this rationale, I store the notes in the corresponding project files (see LINKME) whilst storing the pdfs in a central location, synced with a cloud service provider.

Following are the aspects that I look out for when reading papers:

problem statement
What is the problem the paper is trying to solve? why is this problem important? why should I care? This generally leads to a good material for the introduction of my paper as well.
solution
What is the solution the paper proposes? how is it better/compared to existing ones (do the authors try to show its worth by comparing to existing solutions)? why should I care about it?
results

What were the analytical experiments conducted and what were their results? what datasets were used? what models were used and how did they perform? how were their performance evaluated?

Generally, in my field these questions can be answered by reading the Results section, very rarely are results disclosed in the conclusion section. Note that falls out of the 'first pass' content so I may come back to this only if I need this information for the task at hand.

limitations
Does the paper identify its limitations? what future work can be done?

After writing down notes on the topics above, I try to write a one line summary of the paper along with my remarks. These are always guided by the research question or work at hand. This is also why I prefer not to make dedicated notes.

The "why should I care?" flavor questions will generally be guided by the research question or personal interest that I may have at the time of reading.

Practical recommendations

Here I list some recommendations for cultivating new areas of interests and making a habit of learning through deliberate practise.

  1. Find online resources to read "on the go". By this I mean websites, online communities, blogs, etc which are short articles and can be read on a mobile device.
  2. Find resources (online or otherwise) to study from. For topics which require a sit down, pen and paper, a good book or an online course is required.

Productivity

Productivity to me comes from proper information management. Information can come in many forms: email, thoughts/ideas, deadlines, events, project planning, online links, and many more.

Categories of Information

I categorize information into two broad categories:

Soft
these are bits of information which have a temporary life span. They simply serve as a reminder to do something at a later time or perhaps a scratch list for what groceries to by. Soft information does not require to be stored permanently and can be discarded once the serve their purpose. Soft information does however require that they be captured quickly so that they do not take up space in the limited capacity of our brains. The ability to capture them should also be available to us at all times, under any circumstances, provided we have access to pen+paper or an electronic device such as a smartphone, tablet or computer.
Hard
These are long forms of writing such as notes on matters of interest, project updates and planning, thoughts and ideas worth investigating. Hard information must be searchable (and thus digital) and linkable to one another (so that we can form connections). Hard information should be preserved and must contribute to an ever growing body of knowledge. Finally, hard information should be sharable (in pdf or html) with others (another reason why they should be digital).

First capture then review

I have used many productivity tips, techniques, applications and other forms of nonsense over the years. the one thing which has stuck with me and have proven to work time and time again is the simple, two phase act of:

  1. capturing information as quickly as possible, and
  2. reviewing and progressive summarization of the information later at a designated time.

I think this works for several reasons, first being that it is utterly simple. second, it enforces deliberate practise of sitting down and reflecting, and writing which enriches and exercises the ultimate productivity tool which we possess: our brain.

capturing

I find that absolutely nothing can beat a pen and a paper for this job (see Benefits of writing on paper).

For the rare occasions when pen and paper cannot do the job (such as saving a link to an online article to be read later or when we are on the move), a digital solution is required. Any app will do, as long as it has a "quick capture" mechanism (most todo apps do these days). You can also simply email yourself if you prefer not to have another dependency, another moving part, another "cog" in your system.

Reviewing

Now that I have captured information, what do we do with it? 90% of it is just noise and there is a fine line between expanding knowledge and hoarding knowledge. Here, I find it important to maintain a reference of sort to determine what is important and what is not.

I find that writing on paper acts as a natural buffer to 'noise'.

I review my notes on a weekly, monthly, quarterly and yearly basis. The goal of each review is to progressively summarise the captured ideas. So, weekly reviews summarize daily logs, monthly reviews summarize the weekly reviews, and so on and so forth. This progressive summarization is a deliberate practise of writing that allows me to develop my ideas.

See Benefits of writing on paper.

Benefits of writing on paper

When I am studying or trying to understand a concept, I want free flow, I want independence to sketch, to write text or a mathematical equation. Digital mediums will not work here because they enforce a structure, but at this stage I do not have any structure! I simply have many ideas which I must assimilate and process, arrangement comes later when I review my notes.

Capturing thoughts and ideas on paper also has benefits. It adds the perfect quantity of resistance and acts as a natural buffer to 'noise'.

The migration of an analog note to digital system enforces deliberate progressive summarization of the note.

Notes on setting up website

My thoughts and experiences trying to get my website up and running. The goal was to not use a CMS (content management service), use plaintext to author, and have full control over the services (no paid services).

Jekyll and its problems

Github requires a CNAME file in the directory from which the website should be served. Now, the github-pages gem can be used to instruct Github Pages (GHP) to automatically build and serve the website.

But I faced several challenges getting the compatible versions of the github-pages, jekyll and ruby to match.

So, either I build my website using Jekyll from scratch (back to square one) or I give up this nonsense and just use simple HTML+CSS to build my site (see Notes on responsive web design) using org to write and org-publish to generate the HTML pages (see org-publish).

org-publish

After the numerous hassles with Jekyll (see Jekyll and its problems), I have switched over to org-publish to generate the html and github pages as usual to host them.

I am very pleased with its simplicity and text-centric nature. Also the fact that it just worked out of the box was a pleasant surprise.

Notes on responsive web design

These notes summarize the detailed article on MDN web docs.

There are 4 areas to keep in mind:

  1. responsive content: use flexbox and grid for this (media queries are a thing of the past)
  2. responsive images: always max-width: 100%; so that images scale based on the parent
  3. responsive typography: use a combination of vw and rem units to scale text on screen

  4. viewport meta tag: always set it to the following 

    <meta name="viewport" content="width=device-width,initial-scale=1">

Emacs PATH and exec-path

Emacs maintains its own copy of the PATH variable which it copies over from the shell. However, on OSX it does not copy the paths set by libexe, but only the ones explicitly set in the shell config files.

The exec-path is a list which Emacs uses to look for programs (such as find, grep, etc). Emacs documentation claims to set this up from PATH but this does not seem to be case (at least on OSX).

So the simplest solution I have come across (as of [2020-09-14 Mon 20:27]) is to set them up manually. See aru-path.el for details.

Notes on working with Python in Emacs

I like to use python's builtin virtualenv package to manage virtual environments for my projects. Typically, I create a virtualenv per project and activate it with the source venv/bin/activate shell command (note that I always call my virtualenv venv, since it is inside the project directory, there is no namespace collision).

I like the simplicity of the activate script, if you look at the source code, you find that all it does is append a certain directory to the $PATH environment variable such that the python executable from the virtualenv is picked up by the system.

I dislike tools such as pyenv and pyenv-virtualenv simply because they are external dependencies which I may not have on remote machines and second because they add to my shell's startup time.

In emacs, each buffer has a local value for the default-directory variable, which is set to the directory in which the file exists. This is a very reasonable default and should not be changes to perhaps the project root directory because other commands such as find-file depend on it.

The manual way to start a python process is to open a dired buffer at the project root, and then issuing the run-python command with a universal argument (C-u C-c C-p if you want to do it the keybinding way). This is the simplest solution as it requires zero config on emacs and shell side.

In the future, I would like to automate this, perhaps the functionality can be wrapped in a aru/run-python function.

Notes on working with Ruby

I am aware of chruby and ruby-install projects which allow one to manage and install multiple ruby versions respectively. However, not being a ruby developer, I do not have such a use case and wish to circumvent the complexity that comes with such tools.

My requirements for ruby (as of [2021-01-29 Fri]) is simply to build my personal website which uses Jekyll.

As such, I opt to install ruby via Homebrew. This has a slight drawback that is, it requires me to manually add the location of the 'gems' and the ruby executable to the PATH.

The latter is easily done since brew keeps the latest executable at /usr/local/opt/ruby/bin. The prior requires some more manual updates everytime brew updates ruby since gems are stored at /usr/local/lib/ruby/gems/x.x.x/bin where the 'x's represent the latest version of ruby.

I follow a similar strategy for Python (see ADDME). Also see notes on Emacs PATH and exec-path.

Notes on working with Python

Homebrew provides the usr/local/opt/python@x.x/libexec/bin directory which contains unversioned executables for python and pip which are correctly symlinked to the actual python@x.x executables.

I add this to system and emacs PATH (see Emacs PATH and exec-path). I prefer this to scripts such as pyenv since it adds to my shell startup time.
